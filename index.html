What Is Node.js, and Why Does It Exist?

Node.js is JavaScript running outside a browser. Before Node, JS could only run inside web pages. Node took Chrome's V8 engine and bolted on file access, networking, and more — so JavaScript could also power servers: programs that stay running in the background, waiting to answer requests from other computers (a browser, another app, anything).

🍳 Think of it like this

A restaurant. Your code is the order the customer places. Node's APIs are the standard equipment every kitchen has — an oven, a fridge. V8 is the chef who actually reads the ticket and cooks. libuv is the rest of the kitchen staff — handling anything that takes time, so the chef is never stuck standing still. We'll come back to this kitchen all page long.

Node.js = V8 (runs JS) + libuv (handles everything slow) + a toolbox of APIs on top. Before going deeper into how they work together, let's make this concrete: how do you actually run any of it?

▶️node file.jsCommand›

The one command you'll type more than any other.

What: Runs one JavaScript file, top to bottom, using Node instead of a browser.
Why: Your code needs an engine to execute it — Node is that engine on your machine.
Does: Reads the file, runs your code, then exits once there's nothing left to do (more on exactly when, in the Event Loop section).
Connects: This is genuinely the only command you need for most of what's on this page.

A live prompt for testing one line at a time.

What: Type node alone, with no filename, to get an interactive shell.
Why: Perfect for testing a tiny snippet without creating a whole file.
Does: Runs each line the moment you hit enter, and shows you the result immediately.
Connects: Same engine as running a file — just interactive instead of reading from disk.
🖨️console.log()Output›

Your window into what your code is actually doing.

What: Prints a value to your terminal — Node's version of "let me see what's happening."
Why: There's no browser page to look at — this is how you'll debug almost everything below.
Does: Accepts strings, numbers, objects, arrays — anything — and prints a readable version.
Connects: Formally, this lives on the console global object — see Globals below.

Restarts your server automatically when you save.

What: A small tool that watches your files and restarts Node the moment one changes.
Why: Without it, you'd manually stop and rerun node after every single edit.
Does: Swap node app.js for npx nodemon app.js.
Connects: Installed like any other npm package — see Modules & npm below.

The Event Loop — The Heart of Node.js

Node runs your JS on one single thread. The Event Loop is the trick that lets that one thread handle thousands of things "at once" without ever freezing — by never actually waiting for anything slow.

🍳 Back to the kitchen

The chef (Call Stack) can only do one thing at a time. When an order needs something slow — marinating, waiting on a delivery — the chef hands it to the kitchen staff (libuv) and immediately moves to the next ticket. The staff finishes in the background and drops the result on the pass (the queues). The expediter (Event Loop) has one rule: clear every VIP ticket on the pass (Microtask Queue) before calling out a single regular ticket (Callback Queue).

  1. Sync code runs first. Everything on the Call Stack executes top to bottom, immediately.
  2. Async calls get handed off. `setTimeout`, `fs.readFile`, network calls — these go to libuv / Web APIs instead of blocking the stack. (If "callback" isn't a familiar word yet: it just means a function you hand over to run later. Fully explained next, in Async Patterns.)
  3. Promises jump the line. When a Promise resolves, its `.then()` goes into the Microtask Queue — higher priority than regular async callbacks.
  4. Work finishes in the background. Once a timer fires or a file finishes reading, its callback moves to the Callback Queue.
  5. The Event Loop checks: is the stack empty? If yes — it drains the entire Microtask Queue first, always.
  6. Then it takes ONE callback from the Callback Queue, runs it, and starts the whole check again. Forever, for the life of your program.
🧠Call StackRuntime›

Where your code actually runs, one function at a time.

What: A stack of functions currently executing.
Why: JS is single-threaded — it needs to track "what's running right now."
Does: Pushes a function on when called, pops it off when it returns.
Connects: Must be empty before the Event Loop adds anything new.
🛰️Web / Node APIs (libuv)Runtime›

Background helpers that do the actual waiting for you.

What: Timers, file I/O, and network operations, handled outside the Call Stack.
Why: So one slow file read doesn't freeze your entire app.
Does: Waits in the background, then reports back when done.
Connects: Sends finished work to the Callback Queue.
📥Callback QueueRuntime›

The waiting line for finished timers and I/O callbacks.

What: Also called the "macrotask queue."
Why: Finished async work can't jump straight into a busy Call Stack.
Does: Holds callbacks (setTimeout, I/O, setImmediate) until it's their turn.
Connects: Event Loop pulls ONE of these per cycle — after microtasks.
⚡Microtask QueueRuntime›

A faster, higher-priority line — mostly for Promises.

What: Holds `.then()`, `.catch()`, and `async/await` continuations.
Why: Promise results should run right after current code — before anything else waits.
Does: Gets fully emptied before the Callback Queue gets a single item.
Connects: This is why Promises always "feel faster" than setTimeout(0).
🔁Event LoopRuntime›

The traffic controller deciding what runs next.

What: A loop that never stops running for the life of your program.
Why: Someone has to decide what enters the Call Stack once it's empty.
Does: Checks stack empty → drains microtasks → runs one callback → repeats.
Connects: This single loop is what makes async JavaScript possible at all.
🧵Single-Threaded ArchitectureRuntime›

Node runs your JS on exactly one thread. Always.

What: Only one line of your JavaScript executes at any instant.
Why: Simpler mental model — no race conditions inside your own code.
Does: Forces every slow operation to be handled asynchronously instead.
Connects: The entire reason the Event Loop + async model exists.
🚫🧱Non-Blocking I/ORuntime›

Slow operations never stop the rest of your code.

What: Reading files, calling APIs, querying a DB — none of it freezes your app.
Why: One thread can't afford to sit around waiting.
Does: Hands slow work to libuv, keeps executing other code meanwhile.
Connects: Made possible by libuv + the Event Loop working together.
👷Worker ThreadsRuntime›

Real extra threads — for real CPU-heavy work.

What: Actual parallel threads, separate from the main one.
Why: The Event Loop only fixes I/O waiting — it can't help with heavy computation (image processing, big loops).
Does: Runs expensive JS in parallel, off the main thread, without blocking it.
Connects: The one real exception to "Node is single-threaded."
⏱️Sync vs AsyncRuntime›

Run now and block, or run later and don't.

What: Sync = blocks until finished. Async = keeps going, finishes later.
Why: Async keeps your one thread free while waiting on slow things.
Does: Async work is expressed via callbacks, Promises, or async/await.
Connects: These three are just different syntax for the same underlying Event Loop mechanics.
🔌Program LifecycleRuntime›

Why your program ends when it does — or doesn't.

What: Node keeps the Event Loop spinning as long as something is still pending — an open server, a running timer, an unfinished callback.
Why: Otherwise your program couldn't wait around for a request that hasn't arrived yet.
Does: Once the Call Stack, both queues, and every pending timer/handle are empty, Node exits on its own.
Connects: This is why a plain script with only sync code exits instantly, but a script that calls `http.createServer().listen()` runs forever until you stop it.

Async Patterns — Same Problem, Three Syntaxes

Callbacks, Promises, and async/await all do exactly the same thing under the hood — they just look different in your code.

🍳 Same kitchen ticket, written differently

A callback is scribbling "call me when it's ready" on the ticket. A Promise is a numbered pickup ticket — you'll hear "pending," then "ready" or "sorry, we ran out." async/await is handing someone the ticket and telling them to wait right at the counter instead of wandering off — from the outside, it looks like they never left.

1. Callbacks (original)
callback.js
fs.readFile('file.txt', (err, data) => {
  if (err) throw err;
  console.log(data);
});
2. Promises (cleaner)
promise.js
fs.promises.readFile('file.txt')
  .then(data => console.log(data))
  .catch(err => console.error(err));
3. async/await (reads like sync)
async.js
async function read() {
  try {
    const data = await fs.promises.readFile('file.txt');
    console.log(data);
  } catch (err) { console.error(err); }
}
📞CallbacksOriginal›

"Call this function when you're done."

What: A function passed into another function, to run later.
Why: The earliest way JS handled "do this, then that."
Does: Node convention: `(err, data) => {}` — error always comes first.
Connects: Nested callbacks get messy — Promises exist to fix that.
🤞PromisesModern›

An object representing "a value that will exist eventually."

What: Has 3 states: pending, fulfilled, or rejected.
Why: Flattens nested callbacks into readable `.then()` chains.
Does: Its `.then()`/`.catch()` handlers go into the Microtask Queue.
Connects: `async/await` is just syntax sugar built on top of Promises.
✨async/awaitSugar›

Makes Promise code read like normal, top-to-bottom code.

What: `await` pauses a function until a Promise settles — without blocking the whole thread.
Why: `.then()` chains get hard to read once things get complex.
Does: Under the hood, it's still Promises and the Microtask Queue.
Connects: Only "pauses" the function it's in — the rest of your app keeps running.
🧯Error HandlingSafety›

Every async style has its own way to catch failure.

Callbacks: check `err` as the first argument, every time.
Promises: add a `.catch()` at the end of the chain.
async/await: wrap it in a normal `try/catch` block.
Connects: Uncaught errors can crash the entire Node process — always handle them.
🛟Crash HandlingSafety›

Node's last line of defense for errors that slip through.

What: `process.on('unhandledRejection', ...)` and `process.on('uncaughtException', ...)`.
Why: One Promise nobody caught can otherwise crash your entire server, not just that one request.
Does: Lets you log the error and shut down cleanly instead of Node killing the process with no warning.
Connects: A safety net for when try/catch and `.catch()` get missed — not a replacement for them.
process.on('unhandledRejection', err => { ... });
Now that you can write async code, let's talk about organizing it into files — and reusing code other people already wrote.

Modules & Package Management

Node splits code into files called modules. This is how your code is organized, shared, and reused — including code other people wrote (npm).

package.jsondeclares what packages your project needs
↓ npm install
node_modules/npm downloads the actual code here
↓ require() / import
Your Codepulls in exactly what it needs, when it needs it
📦require()CommonJS›

Node's original way to load another file's code.

What: A function that loads and returns another module's exports.
Why: JS originally had no built-in module system — Node needed one.
Does: Reads the file, runs it once, caches and returns its exports.
Connects: Pairs with `module.exports` on the other end.
const fs = require('fs');
📤module.exportsCommonJS›

How a CommonJS file shares its code with others.

What: An object every file has by default; whatever you attach to it becomes public.
Why: Files need a way to say "here's what other files can use."
Does: Whatever you assign here is what `require()` returns elsewhere.
Connects: The export side of the `require()` pair.
module.exports = { add };
🆕import / exportES Modules›

The modern, official JavaScript module syntax.

What: The standard module system built into the JS language itself.
Why: Browsers and Node needed one shared standard, not just Node's own.
Does: `export` shares code, `import` pulls it in — same idea as CommonJS, different syntax.
Connects: Set `"type": "module"` in package.json to use this instead of CommonJS.
import fs from 'fs';
export const add = ...;
⚖️CommonJS vs ESMCompare›

Two systems, same job, different rules.

CommonJS: `require()`, synchronous, Node's original default, files are `.js` or `.cjs`.
ESM: `import`/`export`, can be async, the modern web standard, files are `.mjs` or `.js` with `"type":"module"`.
Connects: You'll see both in real projects — most new projects lean ESM today.
🧰Built-in ModulesCore›

Tools that ship with Node — zero install needed.

What: `fs`, `http`, `path`, `os`, and dozens more.
Why: Common needs (files, servers, paths) shouldn't require external packages.
Does: Just `require('fs')` — it's already there, part of Node itself.
Connects: Detailed one-by-one in the Core APIs section below.
📮npmTool›

Node's package manager — installs code others wrote.

What: "Node Package Manager" — a command-line tool + a public registry.
Why: Nobody should rebuild what already exists (Express, Lodash, etc).
Does: Downloads packages, tracks versions, runs project scripts.
Connects: Reads/writes `package.json`, fills `node_modules/`.
npm install express
🪪package.jsonConfig›

Your project's ID card.

What: A JSON file describing your project.
Why: npm and Node both need to know: name, version, dependencies, entry point, scripts.
Does: Lists dependencies so anyone can `npm install` and get the exact same setup.
Connects: The source of truth npm reads before touching `node_modules`.
🔒package-lock.jsonConfig›

Pins the exact versions package.json only suggests.

What: An auto-generated file recording the precise version of every installed package, including sub-dependencies.
Why: `"^4.2.0"` in package.json still allows drift — this file pins it exactly, so every install is identical.
Does: Written automatically by `npm install`, read automatically on the next one.
Connects: Commit this to git. Never edit it by hand.
📜npm scriptsTool›

Short names for the terminal commands you run daily.

What: A `"scripts"` block inside package.json, mapping short names to real commands.
Why: So everyone on the project types `npm start`, not a command they have to memorize.
Does: `npm run ` — or the shortcuts `npm start` / `npm test` for the common ones.
Connects: The main entry point most projects are actually launched from.
"scripts": { "start": "node app.js" }
npm start
🚀npxTool›

Runs a package's command without a global install.

What: Downloads-and-runs (or reuses) a package's command, just for one use.
Why: You don't want every one-off tool cluttering your whole system permanently.
Does: `npx nodemon app.js` — no separate `npm install -g` step needed.
Connects: Also what powers generators like `npx create-react-app`.
npx nodemon app.js
🧩dependencies vs devDependenciesConfig›

Two lists, for two different jobs.

What: Two separate arrays inside package.json.
Why: Some packages (Express) run in production; others (a test runner) are only needed while building.
Does: `npm install X` → dependencies. `npm install X --save-dev` → devDependencies, skipped in a production install.
Connects: Keeps your deployed app's `node_modules` smaller and faster to install.
npm install express
npm install --save-dev jest
📁node_modules/Folder›

Where npm actually puts the code it installs.

What: A folder full of every package your project depends on.
Why: `require()`/`import` need the actual files to exist somewhere on disk.
Does: Gets read automatically whenever you import a package by name.
Connects: Never edited by hand, never committed to git — always regenerable from `package.json`.
Once your code has the modules it needs, it uses global objects Node gives every file for free — no import required.

Global Objects & Environment

Node quietly hands every file a few free tools — no `require()` needed.

🖥️consoleGlobal›

The object behind every console.log() you've already typed.

What: A global object for printing to your terminal.
Why: You need some way to see what's happening inside code that isn't visual.
Does: `.log()` for normal output, `.error()` for errors (a separate stream), `.table()` for arrays of objects, and more.
Connects: Available in every file automatically — the first global you actually used, back in Running Your Code.
console.log('debug:', someValue);
⚙️processGlobal›

Represents the currently running Node program.

What: A global object describing this exact running instance of Node.
Why: Your code often needs to know about its own environment (args, env vars, when to exit).
Does: Exposes `process.env`, `process.argv`, `process.exit()`, and more.
Connects: The main gateway to environment variables and system info.
📍__dirname / __filenameGlobal›

Always know exactly where the current file lives.

What: `__dirname` = folder path, `__filename` = full file path.
Why: File paths need to work no matter where someone runs your script from.
Does: Gives an absolute path, safe to build other paths from.
Connects: Commonly combined with the `path` module.
path.join(__dirname, 'data.txt')
🔑Environment VariablesGlobal›

Config that lives outside your code.

What: Key-value settings from the OS or a `.env` file — API keys, ports, secrets.
Why: Secrets and per-environment config shouldn't be hardcoded or committed to git.
Does: Read via `process.env.VARIABLE_NAME`.
Connects: Same code, different config, across dev/staging/production.
const port = process.env.PORT;
Also global, for what it's worth: `Buffer` and the timer functions (`setTimeout`, `setInterval`, `setImmediate`) need no `require()` either, exactly like these. They get their own cards next, in Core APIs, since they're each tied to one specific job.

Core Built-in Modules

These are what your code actually calls day to day — all backed by libuv underneath.

📄fsFile System›

Read and write files on disk.

Does: `readFile`, `writeFile`, `mkdir`, watch for file changes, and more.
Connects: Async by default — runs through libuv's thread pool, not the main thread.
fs.readFile('x.txt', (err, data) => {...})
🌐httpServer›

Build a web server, no framework required.

Does: Creates a server that listens for requests and sends responses.
Connects: Frameworks like Express are built on top of this exact module.
http.createServer((req,res)=>{...}).listen(3000)
🧭pathUtility›

Safely build file paths on any OS.

Does: Joins, resolves, and normalizes paths — handling `/` vs `\` for you.
Connects: Used constantly alongside `fs` and `__dirname`.
path.join('folder', 'file.txt')
💻osSystem Info›

Info about the machine Node is running on.

Does: CPU count, free memory, platform, home directory, and more.
Connects: Handy for scaling decisions (e.g., how many Worker Threads to spawn).
🧱BufferBinary Data›

How Node handles data that isn't plain text.

What: Raw chunks of binary data (images, video, network packets).
Why: JS strings can't safely hold arbitrary binary bytes.
Does: Technically a global — no `require()` needed for everyday use, unlike `fs`/`http`/`path`/`os`.
Connects: This is exactly what flows through Streams.
🌊StreamsData Flow›

Process data in chunks, not all at once.

What: A way to read/write data piece by piece instead of loading it all into memory.
Why: You can't load a 10GB video file into RAM at once.
Does: Emits chunks as `Buffer` objects over time, as events.
Connects: Built directly on top of `EventEmitter`.
📡Events & EventEmitterPattern›

Node's "something happened" pattern.

What: An object you can `.on('event', fn)` and `.emit('event')`.
Why: Many things in Node aren't request/response — they just "happen" over time.
Does: Powers `http` servers, Streams, and most of Node's async APIs under the hood.
emitter.on('data', chunk => ...)
⏰TimersScheduling›

Schedule code to run later.

setTimeout: run once, after at least N milliseconds.
setInterval: run repeatedly, every N milliseconds.
setImmediate: run right after the current I/O phase — often before a 0ms timeout.
Connects: All three are globals (no `require()` needed) and land in the Callback Queue, not the Microtask Queue.

Can You Answer These?

No looking up code — just check whether the mental model stuck. Click a question to reveal the answer.

1. What is the "call stack" in JS?
The list of functions currently being executed, stacked in order of who called whom. When a function returns, it's popped off. If it never returns (an infinite loop), the stack — and the whole thread — is stuck.
2. What's the difference between synchronous and asynchronous code, really?
Sync = each line waits for the previous one to finish before running. Async = a line can start work and let later lines run before that work finishes — the result shows up later via callback/Promise.
3. Why do people say "Node has one call stack but many things happening at once"?
Because the concurrency lives outside JS — in the OS/libuv. Your JS only ever runs one thing at a time on the stack; the "waiting" parts (disk, network, timers) happen elsewhere and just report back.
4. Why is Node good for I/O-heavy apps but bad for CPU-heavy ones?
Its whole design (single thread + event loop + non-blocking I/O) shines when most time is spent waiting (DB calls, network, files). CPU-heavy work has nothing to "wait" on — it just hogs the one thread and blocks everyone else.
5. What's the difference between `null` and `undefined` in JS (comes up constantly in Node)?
`undefined` = a variable exists but was never assigned a value (JS's default). `null` = someone deliberately set "no value here." Node APIs often use `null` intentionally (e.g., `err` is `null` when there's no error).
6. What is `package.json` actually for?
It's the manifest: your project's name, version, dependencies, scripts (`npm run ...`), and entry point. It's how npm and Node know what your project needs and how to run it.
7. What's the difference between a module and a package?
A module is any single file with its own scope (anything you `require`/`import`). A package is a folder with a `package.json` — usually a collection of modules published as one unit (e.g., on npm).
8. `module.exports` vs `exports` — why does `exports.foo = bar` sometimes not work?
`exports` is just a variable that points to the same object as `module.exports`. Adding properties to it works. But the moment you do `exports = something`, you break that link — `module.exports` still points to the original object, and that's what actually gets returned by `require()`.
9. Why doesn't changing a file restart your server unless you use something like nodemon?
Node reads your code once at startup and runs it — it has no built-in file-watching. Tools like nodemon watch the filesystem and just restart the `node` process for you when a file changes.
10. What does `npm install` actually do, step by step?
Reads `package.json` → resolves every dependency (and their dependencies) into a tree → downloads them into `node_modules` → writes/updates `package-lock.json` to lock exact versions.
11. What's the actual role of `package-lock.json` if `package.json` already lists versions?
`package.json` often uses ranges (`^1.2.0`) — flexible. `package-lock.json` pins the exact resolved version of every package and sub-dependency, so `npm install` produces an identical tree on every machine, not just something "compatible."
12. What's the actual difference between `dependencies` and `devDependencies`?
`dependencies` = needed to actually run your app (like `express`). `devDependencies` = only needed while you're developing/testing (like `nodemon`, `jest`).
13. What is `this` inside a regular Node module — what does it point to?
At the top level of a CommonJS module, `this` is `module.exports` (not the global object) — because Node wraps every file in a function behind the scenes before running it.
14. Why does almost every Node callback have the signature `(err, data)`?
It's a convention called "error-first callbacks." Check `err` first — if it's not `null`, something failed and `data` should be ignored. It became the standard before Promises existed, and a lot of core Node APIs still follow it.
15. If two different files both `require('./math.js')`, does that file's code run twice?
No. Node caches the module the first time. Second `require` just reuses the saved result.
16. A plain script with only sync code exits the instant it finishes. A script with `http.createServer().listen()` runs forever. Why the difference?
The plain script finishes its code and has nothing left to do, so it exits. `http.createServer().listen()` keeps an open "listening" handle — Node stays alive waiting for requests.
17. If Node is single-threaded (mostly), how does it handle thousands of concurrent connections?
Because it's never actually blocked waiting. Each request hands its I/O work off (to the OS or the thread pool), registers a callback, and moves on immediately. The single thread just keeps cycling through the event loop picking up finished work — it's concurrency through non-blocking I/O, not through parallel threads.
18. Why doesn't reading a 2GB file freeze the rest of your Node app?
The actual file reading happens in the background (via libuv's thread pool), not on your main JS thread. So your code keeps running while it waits.
19. What's the one real exception to "Node is single-threaded"?
The libuv thread pool — a few background threads (default 4) that handle heavy stuff like file I/O, DNS, crypto.
20. `worker_threads` vs the libuv thread pool — aren't these the same thing?
No. The libuv thread pool is internal and only used by specific built-in APIs (`fs`, `crypto`, `dns`). `worker_threads` gives you real separate JS threads with their own V8 instance and event loop — for running your own CPU-heavy code in parallel, not I/O.
21. Why does one bad synchronous loop freeze an entire Node server, but a bad async call doesn't?
The event loop is one thread. Sync code occupies that thread until it finishes — nothing else, not even other users' requests, can run. Async calls hand the waiting part off elsewhere, so the thread stays free to serve everyone else while it waits.
22. Why do people say "never block the event loop" — what counts as blocking?
Anything synchronous and slow: a big `JSON.parse`, a tight CPU loop, `fs.readFileSync` on a huge file, heavy regex. If it doesn't hand control back to the event loop until it's done, it's blocking — and during that time, zero other requests get served.
23. Are callbacks, Promises, and async/await three different mechanisms, or one mechanism written three ways?
One mechanism, three syntaxes. Promises use the same underlying async machinery as callbacks, and async/await is syntax sugar over Promises — the same Event Loop runs underneath all three.
24. What's the difference between the Callback Queue and the Microtask Queue — and which one always empties first?
The Microtask Queue holds Promise callbacks (`.then`/`.catch`/`await` continuations); the Callback Queue holds timers and I/O callbacks. The Event Loop always fully drains the Microtask Queue before taking even one item from the Callback Queue.
25. What's the difference between `process.nextTick()` and a Promise `.then()`?
Both jump the queue ahead of the callback queue, but `process.nextTick` is a separate, higher-priority queue that fully drains before the microtask (Promise) queue even starts. So order is always: nextTick queue → microtask queue → next callback.
26. The event loop has "phases" — what are they, roughly?
Six main ones, in order: timers (setTimeout/setInterval) → pending callbacks → poll (fetch new I/O events, run their callbacks) → check (setImmediate) → close callbacks. After every phase, Node drains the microtask + nextTick queues before moving to the next phase.
27. `setTimeout(fn, 0)` vs `setImmediate(fn)` — which runs first?
Depends where you call them from. Inside a plain script, it's a coin flip (timer precision). Inside an I/O callback (like a file read), `setImmediate` always wins — because you're already past the poll phase, and check comes right after.
28. What happens if a Promise rejects and nothing ever catches it?
It fires an `unhandledRejection` event. Left unhandled, this can crash the entire Node process — not just the one request that failed.
29. EventEmitter — what problem does it actually solve?
It decouples "something happened" from "what to do about it." Instead of hardcoding a callback into the code that does the work, any part of your app can `.on('event', ...)` and react — this is the backbone under HTTP servers, streams, and most of Node's async I/O objects.
30. Why is a Buffer needed at all — why can't Node just use strings for binary data?
Strings are UTF-16 encoded text in JS engines — not raw bytes. A Buffer is a fixed-length chunk of raw memory outside the V8 heap, so Node can move file/network data around without forcing an encoding decision or bloating the JS heap.
31. What actually is a Stream, in one sentence?
A stream is data handled in chunks over time instead of all at once — so you never need the whole thing in memory before you start working with it.
